## Food Lens (여행자용 AI 식재료 분석기) - PRD

### 1. 프로젝트 개요

#### 1.1. 프로젝트 배경 및 목적

여행 중 낯선 식재료나 음식 앞에서 성분, 알레르기 유발 물질, 영양 정보 등을 파악하는 데 어려움을 겪는 사용자들이 많습니다. 특히, 해외여행 시에는 언어 장벽으로 인해 이러한 정보 접근성이 더욱 제한됩니다. Food Lens는 이러한 문제를 해결하기 위해 사용자가 카메라로 식재료나 음식을 촬영하면, AI를 통해 안전성, 주요 성분, 잠재적 알레르기 유발 물질 등의 정보를 즉각적으로 제공하는 모바일 애플리케이션입니다. 이를 통해 사용자는 안심하고 음식을 선택하고 섭취하며, 건강한 여행 경험을 누릴 수 있도록 지원하는 것을 목적으로 합니다.

#### 1.2. 핵심 가치 제안

- **안전하고 건강한 식생활 지원:** 낯선 식재료의 성분 및 안전 정보를 즉시 제공하여 사용자의 건강과 안전을 최우선으로 합니다.
- **언어 장벽 해소:** AI 기반 분석을 통해 언어에 상관없이 식재료 정보를 이해할 수 있도록 돕습니다.
- **정보 접근성 향상:** 복잡한 정보 검색 없이 카메라 촬영만으로 원하는 식재료 정보를 손쉽게 얻을 수 있습니다.

#### 1.3. 주요 목표

1.  **AI 기반 식재료 분석 정확도 향상:** 주요 식재료 및 음식에 대한 AI 분석 정확도를 90% 이상 달성합니다.
2.  **사용자 경험 극대화:** 직관적이고 간편한 카메라 기반 인터페이스를 통해 사용자의 정보 탐색 시간을 최소화합니다.
3.  **주요 여행 국가 식재료 데이터베이스 구축:** 초기 단계에서 한국, 일본, 중국, 태국, 베트남 등 주요 여행 국가의 식재료 데이터를 우선적으로 확보 및 분석합니다.
4.  **안정적인 서비스 운영:** 피크 타임에도 지연 없는 응답 속도와 높은 가용성을 확보합니다.

### 2. 타겟 사용자 및 사용 사례

#### 2.1. 주요 타겟 사용자 그룹

1.  **해외여행객:** 낯선 환경에서 식당 메뉴, 현지 마트 식재료 등에 대한 정보 파악이 필요한 여행객.
2.  **건강 및 알레르기 민감 사용자:** 특정 성분이나 알레르기 유발 물질에 민감하여 음식 섭취에 신중해야 하는 사용자.
3.  **새로운 음식 경험을 추구하는 사용자:** 다양한 나라의 음식을 접하면서도 성분에 대한 궁금증을 해소하고 싶은 사용자.

#### 2.2. 핵심 사용 시나리오

1.  **여행 중 식당 메뉴 분석:** 해외 식당에서 메뉴판을 보고 낯선 음식이 어떤 재료로 만들어졌는지, 알레르기 유발 성분은 없는지 카메라로 촬영하여 확인하는 시나리오.
2.  **현지 마트 식재료 정보 획득:** 해외 마트에서 구매하고 싶은 식재료의 성분, 유통기한, 조리법 등을 카메라로 촬영하여 안전하게 구매할 수 있도록 정보를 얻는 시나리오.
3.  **개인 건강 관리:** 특정 성분에 대한 알레르기가 있는 사용자가 음식 섭취 전, 해당 음식이 자신의 알레르기 유발 물질을 포함하고 있는지 확인하는 시나리오.
4.  **간편한 정보 탐색:** 복잡한 검색 없이 스마트폰 카메라를 이용해 식재료 정보를 빠르게 파악하고 싶은 사용자.

9.  **영양 성분 정보 표시 (P1)**

#### 3.2. 각 기능의 상세 설명

- **1. 카메라 촬영 및 이미지 전송 (P0)**
  - **설명:** 사용자가 스마트폰 카메라를 통해 식재료 또는 음식 이미지를 촬영할 수 있는 기능을 제공합니다. 촬영된 이미지는 Vertex AI API로 전송됩니다.
  - **세부:**
    - 기본 카메라 기능 (초점, 노출 조절 등) 지원
    - 이미지 자동 전송 또는 사용자 확인 후 전송 옵션 제공
    - 고화질 이미지 지원

- **2. AI 기반 식재료/음식 분석 (P0)**
  - **설명:** 촬영된 이미지를 Vertex AI API에 전송하여 식재료 또는 음식의 종류, 주요 성분, 잠재적 위험 요소 등을 분석합니다.
  - **세부:**
    - 이미지 인식 모델 활용
    - 다양한 식재료 및 가공 식품 분석 기능
    - 분석 결과 응답 시간 최소화

- **3. 분석 결과 표시 (안전성, 성분, 알레르기 정보) (P0)**
  - **설명:** AI 분석 결과를 사용자 친화적인 형태로 화면에 표시합니다. 안전성 평가, 주요 성분 목록, 알레르기 유발 물질 포함 여부 등을 명확하게 보여줍니다.
  - **세부:**
    - 그래픽 및 텍스트를 활용한 직관적인 결과 표시
    - 안전 점수 또는 등급 표시
    - 주요 알레르기 유발 물질 (예: 계란, 우유, 땅콩, 밀 등) 명시

- **4. 사용자 계정 생성 및 로그인 (P1)**
  - **설명:** 사용자가 서비스를 이용하기 위해 계정을 생성하고 로그인할 수 있는 기능을 제공합니다. (선택 사항으로 게스트 모드 고려)
  - **세부:**
    - 이메일/비밀번호 기반 회원가입
    - 소셜 로그인 (Google, Apple 등) 지원 (추후 고려)
    - 비밀번호 찾기/재설정 기능

- **5. 과거 분석 기록 조회 (P1)**
  - **설명:** 사용자가 이전에 분석했던 식재료/음식 정보를 기록으로 저장하고 조회할 수 있는 기능을 제공합니다.
  - **세부:**
    - 분석 날짜, 시간, 결과 정보 저장
    - 기록 검색 및 필터링 기능
    - 개별 기록 삭제 기능

- **6. 분석 결과 공유 기능 (P2)**
  - **설명:** 분석된 식재료/음식 정보를 친구나 가족과 공유할 수 있는 기능을 제공합니다.
  - **세부:**
    - 텍스트, 이미지, 링크 등 다양한 형식으로 공유
    - SNS, 메신저 등 외부 앱으로 공유 지원

- **7. 식재료/음식 정보 상세 보기 (P1)**
  - **설명:** 분석된 식재료/음식에 대한 보다 자세한 정보 (예: 원산지, 보관 방법, 칼로리 등)를 제공합니다.
  - **세부:**
    - AI 분석 결과에 기반한 추가 정보 제공
    - 가능한 경우, 외부 데이터베이스 연동 (추후 고려)

- **8. 알레르기 유발 물질 설정 및 알림 (P1)**
  - **설명:** 사용자가 본인 또는 가족의 알레르기 유발 물질을 사전에 설정하면, 분석 결과에서 해당 물질 포함 시 명확하게 경고하는 기능을 제공합니다.
  - **세부:**
    - 다양한 알레르기 유발 물질 목록 제공
    - 설정된 알레르기 물질 감지 시 시각적/청각적 알림

- **9. 영양 성분 정보 표시 (P1)**
  - **설명:** 분석된 식재료/음식의 주요 영양 성분 (칼로리, 탄수화물, 단백질, 지방 등) 정보를 제공합니다.
  - **세부:**
    - 기본적인 영양 성분 정보 제공
    - 섭취량 대비 영양 성분 비율 표시 (추후 고려)

- **10. 추천 조리법/섭취 방법 제공 (P2)**
  - **설명:** 분석된 식재료를 활용한 간단한 조리법이나 섭취 방법을 추천합니다.
  - **세부:**
    - 레시피 데이터베이스 연동 또는 자체 추천 로직
    - 사용자의 선호도 기반 맞춤 추천 (추후 고려)

- **11. 오프라인 분석 모드 (제한적) (P2)**
  - **설명:** 인터넷 연결이 불안정한 환경에서도 일부 기본 정보 (예: 자주 분석하는 식재료의 정보)를 제공할 수 있도록 제한적인 오프라인 기능을 제공합니다.
  - **세부:**
    - 사전에 다운로드된 데이터 기반 분석
    - 온라인 상태 시 최신 정보 업데이트

- **12. 다국어 지원 (결과 표시) (P1)**
  - **설명:** 분석 결과 및 앱 내 텍스트를 사용자의 기기 언어 또는 사용자가 선택한 언어로 표시합니다.
  - **세부:**
    - 영어, 한국어, 중국어, 일본어 등 주요 여행 언어 지원
    - 사용자 기기 언어 자동 감지

- **13. 사용자 피드백 제출 (P1)**
  - **설명:** 사용자가 분석 결과에 대한 피드백 (정확도, 누락 정보 등)을 제출할 수 있는 기능을 제공하여 서비스 개선에 활용합니다.
  - **세부:**
    - 피드백 제출 양식 제공
    - 버그 신고 및 기능 제안 기능

- **14. 결제 기능 (추후 확장 고려) (P2)**
  - **설명:** 유료 기능 (예: 고급 분석, 광고 제거, 특정 국가 데이터 무제한 이용) 도입 시 필요한 결제 시스템을 구축합니다.
  - **세부:**
    - 인앱 결제 (Apple In-App Purchase) 연동
    - 구독 모델 또는 일회성 구매 모델 고려

#### 3.3. 우선순위 표시

- **P0 (필수):** 프로젝트 론칭 시 반드시 포함되어야 하는 핵심 기능
- **P1 (중요):** 론칭 후 빠른 시일 내에 포함되어야 하는 기능
- **P2 (권장):** 향후 발전 가능성이 있으며, 사용자 피드백 및 시장 상황에 따라 추가될 기능

### 4. 기술 사양

#### 4.1. 프론트엔드

- **언어:** Swift
- **프레임워크/라이브러리:** SwiftUI (UI 개발), Vision (이미지 분석 관련 프레임워크), Core ML (온디바이스 ML 모델 활용 시)

#### 4.2. 백엔드

- **언어:** Python (Cloud Functions를 사용할 경우) 또는 없음 (Serverless)
- **프레임워크:** Firebase Cloud Functions (경량 API Gateway 역할)

#### 4.3. 데이터베이스

- **종류:** Google Firestore (NoSQL)
- **이유:**
  - **IOS 친화적 SDK:** Swift용 Firebase SDK를 통해 백엔드 코드 없이 앱에서 직접 실시간 읽기/쓰기가 가능합니다.
  - **오프라인 지원:** Firestore 캐싱 기능을 통해 인터넷 연결이 끊겨도 앱 사용성을 유지할 수 있습니다.
  - **유연성:** 초기 개발 단계에서 스키마 변경이 자유로워 빠른 이터레이션이 가능합니다.

#### 4.4. 인프라

- **클라우드 플랫폼:** Google Cloud Platform (GCP)
- **주요 서비스:**
  - **Vertex AI:** AI 모델 (이미지 인식, 자연어 처리 등) 배포 및 추론을 위한 핵심 서비스
  - **Cloud Functions / Cloud Run:** 백엔드 로직 및 API 엔드포인트 구현
  - **Firestore / Cloud Storage:** 데이터베이스 및 이미지 저장
  - **Cloud Logging / Cloud Monitoring:** 서비스 모니터링 및 로깅

#### 4.5. 기타 도구

- **API:** Vertex AI Client for Swift (또는 REST API 직접 호출)
- **인증:** Sign in with Apple (iOS 필수), Firebase Auth
- **결제:** StoreKit 2 (In-App Purchase)
- **상태 관리 (iOS):** SwiftUI StateObject / ObservableObject, Combine

## 5. 시스템 아키텍처

### 5.1 백엔드 아키텍처 (Serverless)

- **Mobile-First 접근 방식**:
  - 별도의 Python 서버를 두지 않고, **Firebase Cloud Functions** (Node.js/Python) 또는 **iOS 앱에서 직접 Vertex AI API**를 호출하는 방식을 채택합니다.
  - 민감한 로직(결제 검증 등)만 Cloud Functions에서 처리하고, 단순 데이터 조회는 Firestore Security Rules로 제어합니다.

- **데이터 플로우**:
  1.  **사용자 요청**: iOS 앱에서 사진 촬영.
  2.  **API 호출**: 앱에서 Vertex AI API로 이미지 전송 (URL 또는 Base64).
  3.  **AI 모델**: Gemini 2.0 Flash 모델이 결과(JSON) 반환.
  4.  **로컬 처리**: 앱 내부에서 결과를 파싱하여 결과 화면 노출.
  5.  **이력 저장**: 분석 결과를 Firestore에 비동기로 저장 (백그라운드 처리).

### 5.2 UI/UX 아키텍처

- **컴포넌트 계층 구조**:
  - **Page Components**: 각 페이지의 전체 레이아웃과 주요 섹션을 구성합니다. (예: `HomePage`, `ProfilePage`, `SettingsPage`)
  - **Organism Components**: 여러 Atomic 컴포넌트들을 조합하여 의미 있는 UI 블록을 생성합니다. (예: `Header`, `Footer`, `UserAvatarMenu`, `FoodDetailCard`)

  - **Atom Components**: 가장 기본적인 UI 요소들로, 독립적으로 사용 가능합니다. (예: `Button`, `Input`, `Label`, `Icon`, `Typography`, `Modal`)
  - **Templates/Layouts**: 페이지 전체의 구조를 정의하며, Page 컴포넌트들이 이 템플릿을 활용합니다. (예: `MainLayout`, `AuthLayout`)

  - **Style Guide**: 색상 팔레트, 타이포그래피, 아이콘 세트, 간격 체계(spacing) 등 일관된 시각적 언어를 정의합니다.
  - **Component Library**: 재사용 가능한 UI 컴포넌트들을 정의하고 문서화합니다. (예: React Storybook 활용)
  - **Design Tokens**: 디자인 시스템의 기본 값들을 추상화하여 코드와 디자인 간의 일관성을 유지합니다.
  - **Accessibility Guidelines**: 모든 컴포넌트와 인터랙션이 웹 접근성 표준을 준수하도록 설계합니다.

    - **FAQ 목록 확인**: `FAQ 페이지` (`/faq`)에서 제공되는 질문 목록을 스크롤하거나, 검색 창에 키워드를 입력하여 관련 질문을 찾습니다.
    - **답변 확인**: 찾은 질문을 클릭하여 상세 답변을 확인합니다.
    - **추가 지원 필요시**: FAQ로 해결되지 않을 경우, FAQ 페이지 하단의 "문의하기" 링크를 클릭합니다.

2.  **직접 문의하기 플로우**:
    - **시작**: `FAQ 페이지` (`/faq`) 또는 네비게이션 메뉴에서 "문의하기" (`/contact`)를 선택합니다.
    - **문의 폼 작성**: `문의하기 페이지` (`/contact`)에서 이름, 이메일, 문의 내용(겪고 있는 문제, 요구사항 등)을 상세하게 작성합니다.
    - **문의 제출**: "제출" 버튼을 클릭하여 문의 내용을 백엔드로 전송합니다.
    - **답변 수신**: 등록된 이메일로 문의 내용 접수 확인 메일이 발송되며, 담당자가 확인 후 이메일을 통해 답변을 제공합니다. (처리 시간 안내 포함)

3.  **문제 신고 플로우 (분석 오류 등)**:
    - **시작**: `분석 결과 상세 페이지` (`/results/:analysisId`)에서 분석 결과가 잘못되었거나 오류가 있다고 판단될 때, "결과 신고" 또는 "오류 제보" 버튼을 클릭합니다.
    - **신고 폼 이동**: 간소화된 신고 폼으로 이동하거나, `문의하기 페이지` (`/contact`)에서 "분석 오류 신고" 옵션을 선택합니다.
    - **상세 정보 입력**: 어떤 부분이 잘못되었는지, 어떤 식재료였는지, 언제 발생했는지 등 관련 정보를 상세하게 입력합니다. (분석 ID 자동 포함)
    - **신고 제출**: 신고 내용을 제출하고, 접수 확인 및 처리 과정에 대한 안내를 받습니다.

### 5.5 컴포넌트 상태 관리

- **전역 상태 (Global State)**:
  - **사용자 인증 정보**: 로그인 상태, 사용자 ID, 역할 등 전역적으로 접근해야 하는 사용자 관련 정보는 Context API 또는 Redux/Zustand와 같은 상태 관리 라이브러리를 사용하여 관리합니다.
  - **앱 설정**: 사용자의 언어 설정, 테마 설정 등 앱 전반에 걸쳐 유지되어야 하는 설정 값들을 전역 상태로 관리합니다.
  - **API 호출 상태 (일부)**: 로딩 상태, 에러 메시지 등 전역적으로 표시되어야 하는 API 호출 관련 상태를 관리합니다.

- **로컬 상태 (Local State)**:
  - **UI 엘리먼트 상태**: 각 컴포넌트 내에서만 사용되는 상태(예: 버튼의 활성화/비활성화, 토글 버튼의 on/off 상태, 폼 입력 필드의 값)는 `useState` 훅을 사용하여 컴포넌트 내에서 직접 관리합니다.
  - **폼 상태**: 복잡한 폼의 경우, 각 입력 필드의 상태를 관리하고 폼 제출 시 유효성 검사를 수행합니다.
  - **모달/팝업 상태**: 모달 창의 열림/닫힘 상태, 모달 내 데이터 상태 등을 해당 모달 컴포넌트의 로컬 상태로 관리합니다.

- **컴포넌트 상태 정의 예시**:
  - **버튼**: `disabled` (boolean), `loading` (boolean), `variant` (string: 'primary', 'secondary'), `onClick` (function).
  - **폼 입력 필드**: `value` (string/number), `onChange` (function), `error` (string | null), `isTouched` (boolean), `label` (string).
  - **모달**: `isOpen` (boolean), `onClose` (function), `title` (string), `content` (ReactNode), `actions` (ReactNode).
  - **전역 사용자 인증 상태**: `isAuthenticated` (boolean), `user` (object | null), `token` (string | null), `login` (function), `logout` (function).

## 6. 보안 및 인증

- **인증 방식**:
  - **JWT (JSON Web Token)**: 사용자 로그인 성공 시, 서버는 JWT를 발급하여 클라이언트에 전달합니다. 클라이언트는 이 JWT를 HTTP 요청의 Authorization 헤더에 포함하여 서버에 전송합니다. 서버는 JWT의 유효성을 검증하여 사용자를 인증합니다. 토큰은 적절한 만료 시간을 설정하고, 안전한 저장을 위해 HttpOnly 쿠키 사용을 고려합니다.
  - **OAuth 2.0**: Google, Apple 등 외부 소셜 로그인 서비스를 통해 사용자를 인증합니다. 이를 통해 사용자는 별도의 계정을 생성하지 않고도 편리하게 서비스에 접근할 수 있습니다.

- **권한 관리 (RBAC - Role-Based Access Control)**:
  - **역할 정의**: 일반 사용자(User), 관리자(Admin) 등의 역할을 정의합니다.
  - **권한 할당**: 각 역할에 따라 접근 가능한 API 엔드포인트 및 기능에 대한 권한을 부여합니다. 예를 들어, 관리자만 사용자 목록 조회, 서비스 설정 변경 등의 권한을 가질 수 있습니다.
  - **인증/인가 미들웨어**: API 요청 시 JWT를 검증하는 인증 미들웨어와 함께, 해당 사용자의 역할에 따라 요청된 리소스에 접근할 권한이 있는지 확인하는 인가 미들웨어를 적용합니다.

- **데이터 암호화**:
  - **전송 중 데이터 암호화**: 모든 클라이언트-서버 통신은 TLS/SSL(HTTPS) 프로토콜을 사용하여 암호화하여 데이터가 전송 중에 가로채지거나 변조되지 않도록 합니다.
  - **저장 데이터 암호화**: 사용자 비밀번호는 bcrypt와 같은 강력한 해싱 알고리즘을 사용하여 저장합니다. 민감한 개인 정보(예: 결제 정보)는 데이터베이스 저장 시 추가적인 암호화를 적용하는 것을 고려합니다.

- **API 보안**:
  - **Rate Limiting**: 악의적인 공격이나 과도한 요청으로부터 서버를 보호하기 위해 API 엔드포인트별로 요청 빈도를 제한합니다.
  - **입력값 검증**: 클라이언트로부터 전달받는 모든 입력값은 서버 측에서 철저하게 검증하여 SQL Injection, XSS(Cross-Site Scripting) 등의 공격을 방지합니다.
  - **CSRF(Cross-Site Request Forgery) 방지**: 사용자 세션을 이용하는 웹 애플리케이션의 경우, CSRF 토큰을 사용하여 사용자가 의도하지 않은 요청을 수행하는 것을 방지합니다.
  - **보안 헤더 적용**: Content Security Policy(CSP), X-Content-Type-Options, X-Frame-Options 등 보안 관련 HTTP 헤더를 설정하여 브라우저 레벨의 보안을 강화합니다.

## 7. 비기능적 요구사항 (NFR)

- **성능**:
  - **응답 시간**:
    - 일반적인 API 요청(예: 프로필 정보 조회)에 대한 평균 응답 시간: **200ms 이하**
    - 식재료 이미지 분석 완료까지의 최대 시간: **5초 이하** (AI 모델 처리 시간 및 네트워크 지연 포함)
    - 페이지 로딩 시간 (Initial Load): **3초 이하** (First Contentful Paint 기준)
  - **처리량 (Throughput)**:
    - 초당 API 요청 처리 능력: **500 RPS (Requests Per Second)** 이상 (안정적인 서비스 운영을 위한 목표치)
    - 이미지 분석 동시 처리 요청 수: **100 TPS (Transactions Per Second)** 이상

- **확장성**:
  - **사용자 수**: 초기 10만 명 이상의 사용자를 수용할 수 있도록 설계하며, 향후 **수백만 명 이상의 사용자 증가**에 유연하게 대응할 수 있는 아키텍처를 구축합니다.
  - **트래픽 대응**: 트래픽 증가 시, 컴퓨팅 자원(CPU, 메모리), 스토리지, 데이터베이스 용량을 자동으로 확장(Auto-Scaling)할 수 있는 클라우드 기반 인프라를 활용합니다. 마이크로서비스 아키텍처 도입을 통해 개별 서비스의 독립적인 확장을 지원합니다.

- **가용성 (Availability)**:
  - **Uptime 목표**: **99.9%** (연간 약 8.76시간의 다운타임 허용)
  - 이를 위해 데이터베이스 이중화, 다중 AZ(Availability Zone) 배포, 로드 밸런싱, 자동 장애 복구 메커니즘 등을 구현합니다.

- **유지보수성**:
  - **코드 품질**:
    - **코드 리뷰**: 모든 코드 변경 사항은 동료 리뷰를 거쳐 승인받습니다.
    - **코딩 컨벤션**: 일관된 코딩 스타일 가이드라인을 준수합니다.
    - **정적 분석 도구**: ESLint, Prettier 등 정적 분석 도구를 사용하여 코드 오류 및 스타일 문제를 사전에 감지합니다.
  - **테스트 커버리지**:
    - **단위 테스트**: 비즈니스 로직의 핵심 부분에 대해 **80% 이상의 단위 테스트 커버리지**를 확보합니다.
    - **통합 테스트**: 주요 서비스 간의 연동 및 API 엔드포인트에 대한 **70% 이상의 통합 테스트 커버리지**를 확보합니다.
    - **E2E 테스트**: 핵심 사용자 플로우에 대한 End-to-End 테스트를 자동화하여 배포 전 안정성을 검증합니다.
  - **문서화**: 아키텍처, API 명세, 코드 주석 등을 최신 상태로 유지하여 팀원 간의 이해도를 높이고 신규 개발자의 온보딩을 용이하게 합니다.
  - **모니터링 및 로깅**: 애플리케이션의 성능, 오류, 사용량 등을 실시간으로 모니터링하고, 상세한 로그를 기록하여 문제 발생 시 신속하게 원인을 파악하고 해결할 수 있도록 합니다.
